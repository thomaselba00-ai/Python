Before we start

What even is "Object-Oriented Programming"?

Forget the scary name for a second. It's just a way of organizing code so it mirrors real life — as things ("objects") that have properties and can do stuff, instead of code as one long list of instructions.

🏠 The blueprint analogy

Think of a Class as a blueprint for a house — it defines what every house built from it will have (walls, doors, a kitchen) and can do (open a door, turn on lights). An Object is an actual house built from that blueprint. You can build many houses (objects) from one blueprint (class), and each one can have its own paint color, furniture, and owner — its own data — while sharing the same basic design.

One class → many independent objects, each with its own data

Everything below — Encapsulation, Abstraction, Inheritance, and Polymorphism — are just four rules for how these "blueprints" and "objects" are built and behave. Let's go one at a time.

1ENCAPSULATION

Bundle data + behavior, and hide what's private

Encapsulation means packing related information (data) and the actions that use it (methods) into one single unit — the object — and hiding the inner details the outside world doesn't need to see or touch directly.

💊 The medicine capsule analogy

A capsule pill hides its powder inside a shell — you swallow the whole capsule, you don't handle the loose powder directly. A bank ATM works the same way: you press buttons to withdraw money, but you never touch the bank's database directly. The machine exposes safe buttons and hides the risky internal wiring. That's encapsulation: controlled access through a clean interface, with the messy or sensitive parts locked away inside.

Private data stays locked inside · public methods are the only "doors" in

👩‍💻 Code example (Python)

class BankAccount:
    def __init__(self, owner, starting_balance):
        self.owner = owner
        self.__balance = starting_balance   # "__" hides it from outside

    def deposit(self, amount):
        self.__balance += amount
        print(f"Deposited {amount}. New balance: {self.__balance}")

    def withdraw(self, amount):
        if amount > self.__balance:
            print("Not enough money!")
        else:
            self.__balance -= amount
            print(f"Withdrew {amount}. New balance: {self.__balance}")

# --- using the object ---
acc = BankAccount("Riya", 1000)
acc.deposit(500)        # ✅ works through the public "door"
acc.withdraw(2000)       # ✅ safely rejected inside the class
# acc.__balance         ❌ this would fail — balance is hidden

Why it matters

Without encapsulation, any part of a huge program could carelessly change your balance directly, skip safety checks, and create bugs that are almost impossible to trace. Hiding data and forcing access through controlled methods keeps data safe and consistent.

2ABSTRACTION

Show only what matters, hide the complex machinery

Abstraction means giving people a simple way to use something, while hiding the complicated logic that makes it actually work underneath.

🧊 The iceberg analogy

When you drive a car, you use a steering wheel, an accelerator, and a brake. You don't need to know how fuel injection, combustion, or the transmission works internally. That whole engine is the "underwater" part of the iceberg — hidden complexity. The steering wheel and pedals are the small visible tip — a simple interface anyone can use.

The user only ever touches the visible tip — the engine stays hidden below

👩‍💻 Code example (Python)

class Car:
    def drive(self):
        self.__start_fuel_injection()   # hidden step
        self.__ignite_combustion()      # hidden step
        self.__engage_gearbox()         # hidden step
        print("🚗 Car is moving!")

    def __start_fuel_injection(self):
        pass   # complex internal detail

    def __ignite_combustion(self):
        pass   # complex internal detail

    def __engage_gearbox(self):
        pass   # complex internal detail

# --- using the object ---
my_car = Car()
my_car.drive()   # ✅ one simple call — all complexity hidden

Why it matters

Abstraction reduces mental load. A programmer using someone else's code (or you, using your own code six months later) shouldn't need to relearn every internal detail — just the simple, documented way to use it.

🔍 Encapsulation vs. Abstraction — the classic mix-up

Encapsulation is about hiding data and bundling it with methods (a security/design mechanism). Abstraction is about hiding complexity and showing a simple interface (a design goal). Encapsulation is how you build the capsule; abstraction is why — so users only see what matters.

3INHERITANCE

Let a new class reuse an existing one's blueprint

Inheritance lets one class (the child) automatically get all the properties and methods of another class (the parent) — and then add or change a few things of its own.

👪 The family trait analogy

You inherit your general traits — you have eyes, a name, the ability to walk — from being human. But you also add your own specific traits: your particular hobbies or your job. In code, an "Animal" class can define general things (all animals eat and sleep), and a "Dog" class can inherit all of that automatically, adding only what's unique to dogs (barking, fetching).

Dog and Cat both inherit eat() & sleep() from Animal, then add their own move

👩‍💻 Code example (Python)

class Animal:                     # parent / base class
    def __init__(self, name):
        self.name = name

    def eat(self):
        print(f"{self.name} is eating.")

    def sleep(self):
        print(f"{self.name} is sleeping.")

class Dog(Animal):                # child class — inherits from Animal
    def bark(self):
        print(f"{self.name} says: Woof!")

# --- using the objects ---
d = Dog("Bruno")
d.eat()     # ✅ inherited from Animal, no need to rewrite it
d.bark()    # ✅ Dog's own extra behaviour

Why it matters

Inheritance avoids repeating the same code for every similar class. Write shared logic once in the parent, and every child gets it for free — while still being able to specialize.

4POLYMORPHISM

One command, many different behaviors

Polymorphism ("many forms") means the same method name can behave differently depending on which object calls it — no need to remember a different command for every type.

🔊 The "make a sound" analogy

Imagine you shout the same instruction — "make a sound!" — to a dog, a cat, and a duck. Each one responds in its own way: the dog barks, the cat meows, the duck quacks. You didn't need a different instruction for each animal — same command, different result, because each object knows how to handle it in its own way.

Same method name — each object fills in its own version of the behavior

👩‍💻 Code example (Python)

class Dog:
    def make_sound(self):
        print("Woof!")

class Cat:
    def make_sound(self):
        print("Meow!")

class Duck:
    def make_sound(self):
        print("Quack!")

# --- using the objects ---
animals = [Dog(), Cat(), Duck()]

for animal in animals:
    animal.make_sound()   # ✅ same call, different behavior each time
# Output:
# Woof!
# Meow!
# Quack!

Why it matters

Polymorphism lets you write one general piece of code (like the for loop above) that works correctly for many different object types — today's classes and future ones you haven't even written yet — without a single if/else to check "what type is this?".

Test yourself
You want to stop other parts of a program from directly changing a variable. Which pillar is this?
Encapsulation — hide the data (e.g. make it private) and only allow changes through controlled methods like deposit()/withdraw().
A "Manager" class and an "Employee" class share a lot of common fields like name and salary. What should you use to avoid repeating that code?
Inheritance — put the shared fields/methods in a parent class (e.g. Staff) and let both Manager and Employee inherit from it.
You call shape.area() on a list containing circles, squares, and triangles, and each returns the correct area for its own shape. What pillar is this?
Polymorphism — the same method call, area(), produces different, shape-specific behavior for each object.
A TV remote lets you change channels without knowing anything about the antenna, signal decoding, or circuit boards inside. What pillar is this?
Abstraction — a simple, easy interface (the remote's buttons) hides all of the complicated internal machinery.
